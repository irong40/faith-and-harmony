<objective>
Convert the hero background-image from a CSS url() declaration to an HTML img element with fetchpriority="high", and add width, height, and loading="lazy" to all remaining landing page images.

Purpose: CSS background-images are invisible to search engine crawlers and cannot be preloaded efficiently. An img element with fetchpriority="high" tells the browser to fetch it before most other resources, improving Largest Contentful Paint. Below-fold images with loading="lazy" defer their network requests until the user scrolls toward them.
Output: Updated LandingPage.tsx with a hero img element and full attribute coverage on all images. Updated landing.css with the background-image url() removed from the hero rule.
</objective>

    ```css
    background:
      linear-gradient(180deg, rgba(5, 5, 5, 0.85) 0%, rgba(28, 28, 28, 0.9) 50%, rgba(5, 5, 5, 0.95) 100%),
      radial-gradient(ellipse at 50% 0%, rgba(255, 107, 53, 0.15) 0%, transparent 60%),

    ```

    The dark gradients remain. The img element added in Part B will provide the background photo.

    Add a new CSS rule below the existing .lp-hero block to position the new hero img as an absolute background layer:
    ```css
    .landing-page .lp-hero-bg-img {
      position: absolute;
      inset: 0;
      width: 100%;
      height: 100%;
      object-fit: cover;
      object-position: center;
      z-index: 0;
    }
    ```

    The existing `.lp-hero` already has `position: relative` and `overflow: hidden`, so the absolute child will be clipped correctly. The gradient background layers in the updated .lp-hero rule will paint over the img because the gradients are in the CSS background of the parent (they render above the img child in stacking order by default). To ensure the gradient overlays sit above the img, the .lp-hero::before pseudo-element and the .lp-container inside the hero already use `z-index: 1`, and the img will be `z-index: 0`. This is the correct stacking order.
