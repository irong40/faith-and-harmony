<task type="auto">
  <name>Task 2: Balance invoice edge function with orphan prevention</name>
  <files>supabase/functions/create-balance-invoice/index.ts</files>
  <action>
Create the create-balance-invoice edge function modeled on create-deposit-invoice but with these critical differences:

1. Accept { job_id } in POST body (not quote_id, since balance invoicing happens from the job context).

2. Look up the drone_job by job_id, selecting id, quote_id, preview_urls, status. Guard: job status must be 'complete' (processing finished, ready for billing).

3. Look up the quote via job.quote_id, selecting id, total, deposit_amount, quote_requests(name, email, job_type).

4. Calculate balance: balanceAmount = Number(quote.total) - Number(quote.deposit_amount). Convert to cents for Square.

5. ORPHAN PREVENTION (critical difference from deposit function): Insert the payments row with status 'pending', payment_type 'balance', job_id set, square_invoice_id as NULL, due_date as futureDateStr(15) (Net 15) BEFORE calling the Square API. If Square fails, delete the pending row and return error.

6. Create Square invoice with delivery_method "SHARE_MANUALLY" (prevents Square from emailing the client, since we send our own branded email). Invoice number format: SAI-BAL-{shortId}. Title: "Sentinel Aerial Inspections - Balance Invoice". Description referencing job type and short ID.

7. Publish the Square invoice, then update the payments row with square_invoice_id and square_invoice_url from the published response.

8. After successful publish, call the send-balance-due-email edge function using fetch with SUPABASE_SERVICE_ROLE_KEY authorization, passing { job_id, payment_id, square_invoice_url }.

9. Idempotency guard at the top: check for existing balance payment for the same quote_id. Return 409 if found.

Use the same CORS headers, env vars, error handling, and Deno patterns from create-deposit-invoice.
  </action>
