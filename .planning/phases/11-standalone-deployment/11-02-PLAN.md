Key facts from research:
- create-deposit-invoice already supports `SQUARE_ENVIRONMENT` env var toggle (line 14 of index.ts)
- square-webhook validates HMAC using SQUARE_WEBHOOK_SIGNATURE_KEY and SQUARE_WEBHOOK_URL
- WEBHOOK_URL fallback: `${SUPABASE_URL}/functions/v1/square-webhook`
- Sandbox and production are completely separate Square environments. Sandbox invoice IDs have no meaning in production.
- The payments table may contain sandbox records that should be archived before cutover.

<interfaces>
From supabase/functions/square-webhook/index.ts:
- Reads: SQUARE_WEBHOOK_SIGNATURE_KEY, SQUARE_WEBHOOK_URL, SUPABASE_URL, SUPABASE_SERVICE_ROLE_KEY
- Validates HMAC-SHA256 signature using webhookUrl + rawBody
- Processes invoice.payment_made events, updates payments table status to "paid"

From supabase/functions/create-deposit-invoice/index.ts:
- Reads: SQUARE_ENVIRONMENT (default "sandbox"), SQUARE_ACCESS_TOKEN, SQUARE_LOCATION_ID
- Uses connect.squareup.com for production, connect.squareupsandbox.com for sandbox
</interfaces>

**Step B: Register Production Webhook**

- SQUARE_ACCESS_TOKEN = production access token from Step A
- SQUARE_LOCATION_ID = production location ID from Step A
- SQUARE_ENVIRONMENT = production
- SQUARE_WEBHOOK_SIGNATURE_KEY = webhook signature key from Step B
- SQUARE_WEBHOOK_URL = https://qjpujskwqaehxnqypxzu.supabase.co/functions/v1/square-webhook

CLI alternative (run each separately):
